# 📘 プロジェクト全体計画書

## 👥 開発チーム体制（2名）
- **フロントエンド担当（UI＋QA含む）**
  - React + TypeScript + Material UI / Tailwind CSS
  - UI設計、履歴管理UI、多言語対応UI
  - QAテスト（Cypress / Jest）
- **バックエンド担当（全般対応）**

## 🔄 開発フロー（2名体制）
1. **週次スプリント計画**
   - 月曜にタスク分担（フロント vs バック）
   - 金曜に成果物レビュー＋次週計画
2. **並行作業ルール**
   - フロントは「UIモック → API接続 → QAテスト」
   - バックは「APIモック → 本実装 → QAスコア返却」
   - フロントは常にバックのAPIをモックで先行利用
3. **結合テスト**
   - 各フェーズ終了時に結合テスト（UI＋API）
   - QAはフロント側で自動テスト
4. **CI/CDパイプライン**
   - GitHub Actionsで毎プッシュ時にビルド＋テスト
   - Dockerでローカルとクラウド環境を統一

## 🧪 ユーザーテスト計画

### 🎯 テスト目的
- **UI/UX検証:** 推奨強調型UIが直感的に使えるか  
- **機能検証:** アップロード → 生成 → ZIPダウンロード → 履歴保存 → 多言語切替が問題なく動作するか  
- **品質検証:** 生成画像の自然さ、QAスコアの信頼性  
- **国際対応検証:** 日本語 🇯🇵 / 英語 🇺🇸 / 中国語繁体字 🇹🇼 の切替がスムーズか  

### 👥 テスト対象ユーザー
- **社内テスト（α版）:** 開発メンバー＋デザイナー（5〜10名）  
- **外部テスト（β版）:** 広告代理店担当者、デザイナー、マーケター（15〜20名）  
- **国際テスト:** 英語・繁体字中国語ユーザー（5〜10名）  

### 📋 テスト項目
1. 機能テスト  
   - アップロード、排他ルール、バッチ処理、ZIP DL、履歴保存  
2. UI/UXテスト  
   - 推奨フォント・色パレット選択、エラー表示、多言語切替  
3. 品質テスト  
   - 生成画像の自然さ、QAスコアの一致度  

### 🛠 テスト方法
- シナリオテスト（例: 広告を10枚生成してZIPでDL）  
- アンケート（操作性・品質・多言語対応を5段階評価）  
- ログ収集（エラー率、処理時間、DL成功率）  
- フィードバックセッション（βテスト後にヒアリング）  

### 📆 テストスケジュール
- Week 1–2: αテスト（社内） → バグ修正  
- Week 3–4: βテスト（外部） → UI改善・品質調整  
- Week 5: 国際テスト（英語・繁体字中国語） → 翻訳・フォント調整  
- Week 6: 総合レビュー → リリース準備  

### 🌟 成果
- ユーザーからのフィードバックを反映しUI/UX改善  
- QAスコアの信頼性検証  
- 国際対応の完成度確認  
- リリース前に主要バグ解消
